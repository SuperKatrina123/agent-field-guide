# 05 上下文装配与路由

> [!note] 课程说明
> **学习目标**：把前一章的“结构设计”推进到运行时，理解上下文在任务开始、中途推进和收束阶段是如何被装配、裁剪、更新和路由的。  
> **前置知识**：建议先读完 [[03-上下文工程总论]] 和 [[04-上下文结构设计]]。  
> **预计时间**：核心阅读 `55-75 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`任务开始时该装什么`、`中途如何增删改信息`、`不同来源的信息如何路由进入上下文`、`如何在有限预算下保持高信息密度`。

---

> [!question] 带着问题阅读
> 为什么很多 Agent 在前几轮还算正常，但一旦任务继续推进，系统就开始把无关历史带进来、把关键信息冲掉、把新结果和旧状态混在一起？如果结构已经设计好了，运行时到底还会错在哪？

## 1. 第 5 章讨论的不是“结构”，而是“运行时”

第 4 章解决的是静态问题：信息应该怎么长。

你已经知道：

- 上下文需要分层
- System Prompt 该做什么、不该做什么
- 状态最好字段化
- 上下文最好按对象和接口管理

但这还只是设计图纸。

真正让 Agent 表现稳定与否的，往往是运行时问题：

- 这一轮到底装了哪些信息
- 某段历史为什么被带进来
- 某个工具结果为什么没有被带进来
- 某条旧约束为什么还残留着
- 为什么明明结构设计是对的，当前轮判断却还是失真

> [!abstract] 定义
> 本章所说的“上下文装配”，指的是在每一轮调用发生前，系统如何把不同来源、不同层级、不同生命周期的信息组织成当前轮可见上下文的过程；“路由”则指这些信息是否应该进入，以及进入哪一层、以什么形式进入。

## 2. 一个最小装配视角：不是拼接，而是选择

很多实现里，上下文装配看起来像一个简单动作：

- 拼上系统提示
- 拼上任务说明
- 拼上对话历史
- 拼上工具结果

但如果只把它理解成“拼接”，你会忽略真正困难的部分。

对 Agent 来说，装配更像一套选择系统：

- 选哪些信息进入当前轮
- 选这些信息以什么形态进入
- 选哪些信息保留原文
- 选哪些信息只保留摘要
- 选哪些信息完全不进入

所以，装配的本质不是“尽可能多拿”，而是“在当前轮做正确选择”。

> [!tip] 原则
> 运行时装配的核心，不是组装能力，而是筛选能力。

## 3. 一轮调用开始前，系统通常要装什么

虽然不同 Agent 形态差异很大，但对大多数任务型 Agent 来说，一轮调用开始前常见的装配对象通常可以归纳为五类。

### 3.1 固定规则

这一层通常来自固定层，包括：

- System Prompt
- 不可违反的原则
- 高优先级行为边界

这部分一般默认进入，但不意味着每轮都应该无限扩张。

### 3.2 当前任务定义

这一层回答：

- 这一轮到底在解决什么任务
- 当前子任务是什么
- 当前完成标准是什么

没有这一层，模型很容易把本轮动作和全局目标脱节。

### 3.3 当前状态摘要

这一层回答：

- 现在进行到哪一步
- 已确认哪些事实
- 当前还缺什么
- 当前轮最值得关注的决策点是什么

状态摘要通常比原始历史更重要，因为它能把“已经发生的很多事”压缩成“当前轮需要知道的事”。

### 3.4 局部证据

这一层包括：

- 当前轮相关历史
- 刚返回的工具结果
- 当前轮检索结果
- 某个子问题所需材料

这里最重要的不是“都放进去”，而是“只放对当前决策有价值的局部证据”。

### 3.5 当前用户输入

用户输入常常是触发装配变化的起点。

但在复杂任务里，用户输入不是唯一中心。它要被放回：

- 当前任务定义
- 当前状态
- 当前约束

里一起理解，才不容易被系统误读。

```mermaid

    A["固定规则"] --> X["当前轮上下文"]
    B["当前任务定义"] --> X
    C["当前状态摘要"] --> X
    D["局部证据"] --> X
    E["当前用户输入"] --> X
```

## 4. 任务刚开始时，应该如何装配

任务开始阶段的装配，重点不是“信息充分”，而是“方向清楚”。

这一阶段最需要的是：

- 稳定边界
- 明确目标
- 初始状态
- 必要约束

### 4.1 初始轮最怕什么

任务第一轮最常见的问题有两个：

- 目标模糊
- 带入了过多背景

目标模糊会让模型一开始就不知道应该聚焦哪里。

背景过多则会让系统在还没确定问题之前，就被各种不必要材料淹没。

### 4.2 初始轮装配原则

对大多数任务型 Agent，可以先遵守这条简单原则：

> 起始轮优先保证目标和边界清楚，不急着把所有可能有用的信息都带进来。

因为任务刚开始时，很多信息是否相关，其实还没有到能被稳定判断的时候。

## 5. 任务推进中，装配逻辑为什么必须变化

一个典型误区是：上下文模板一旦定了，后面只要继续追加就行。

这会直接导致上下文逐轮失真。

因为任务推进过程中，系统面对的问题会发生变化：

- 一开始需要理解任务
- 中间需要推进步骤
- 后面需要收束和确认结果

不同阶段需要的信息并不相同。

### 5.1 中途阶段更需要状态，而不是全部历史

任务进入中途后，原始历史的价值通常会下降，状态摘要的价值会快速上升。

原因是：

- 决策重点不再是“发生过什么”
- 而是“当前推进到哪、下一步该做什么”

如果这时还在机械带完整历史，系统很容易过载。

### 5.2 收束阶段更需要完成标准

当任务接近结束时，最重要的问题不再是“还有没有更多信息”，而是：

- 是否已经满足完成条件
- 是否还存在关键未确认项
- 输出是否满足任务要求

如果这时上下文仍然围绕材料堆砌，而不是围绕完成标准组织，系统就很容易：

- 无限继续
- 过度搜集
- 明明该结束却继续绕圈

## 6. 中途如何追加、裁剪、摘要和重写

运行时装配不是只会“加内容”，它至少有四种基本操作。

### 6.1 追加

追加适用于：

- 新工具结果出现
- 新用户约束出现
- 新检索结果返回
- 任务状态发生显著变化

但追加的关键不是“原样追加”，而是判断它应该进入哪一层。

### 6.2 裁剪

裁剪适用于：

- 与当前轮无关的旧历史
- 已经失效的局部材料
- 信息量大但信号弱的输出

裁剪不是删掉重要信息，而是主动降低噪音。

### 6.3 摘要

摘要适用于：

- 长历史
- 多轮工具结果
- 已结束但仍需保留其结论的过程

摘要的价值在于用更小体积保留更高密度的可决策信息。

### 6.4 重写

重写适用于：

- 原始工具结果过长
- 外部文档原文不适合直接喂给模型
- 历史表达方式不利于当前轮推理

重写的目标不是美化文字，而是把信息转换成更适合当前决策的表达形态。

> [!info] 方法
> 运行时最有价值的问题不是“这段信息要不要进”，而是“它以原文、摘要还是结构化字段的方式进”。

## 7. 检索、历史、工具结果应该怎么路由

路由问题，本质上是在回答：**不同来源的信息，应该进不进、何时进、进到哪一层。**

### 7.1 历史不是默认全带

历史常见的错误路由方式是：

- 默认把最近 N 轮全部带入

这在简单聊天里有时可用，但在复杂 Agent 里很容易失真。因为最近 N 轮未必是最相关的 N 轮。

更合理的路由思路是：

- 只带与当前子问题相关的历史
- 把更早但更关键的信息提炼成状态摘要
- 把不再相关的局部对话直接排除

### 7.2 检索结果不是查到就喂

检索结果常见的错误是：

- 查到了就全塞进上下文

问题在于：

- 检索召回不等于当前决策可用
- 原始片段常常太长
- 多条结果之间可能冲突

所以检索结果通常要经过至少一轮路由判断：

- 这批结果是否真的和当前决策相关
- 哪几条优先级更高
- 是否应该先压缩再进入

### 7.3 工具结果不是原样追加

工具结果是最常被误处理的一类信息。

很多系统会直接把工具返回原样塞回上下文，这会导致两个问题：

- token 暴涨
- 真正关键信号被埋在低价值细节里

更好的路由方式通常是：

- 原始结果保存在外部
- 当前轮只带关键结论、关键信号和必要证据

### 7.4 约束信息的路由优先级通常更高

相比大量背景材料，约束信息更应该优先被稳定带入。

因为对 Agent 来说，**少知道一点背景，通常比违反关键约束的代价更低。**

## 8. 上下文预算和优先级管理

只要是现实系统，就不可能无限带信息。

预算管理本质上是在回答：

- 当前轮最值得花 token 的是什么
- 哪类信息必须原样保留
- 哪类信息可以摘要
- 哪类信息应当外置

### 8.1 一个实用优先级思路

可以先用下面这类粗粒度优先级：

1. 高优先级：目标、约束、当前关键状态
2. 中优先级：当前轮决策直接相关的证据
3. 低优先级：背景材料、历史细节、辅助性参考

这不是固定公式，但它有一个很重要的设计倾向：

> 越接近当前决策、越影响边界判断的信息，优先级越高。

### 8.2 预算不足时该牺牲什么

预算紧张时，通常更应该优先牺牲：

- 冗长背景
- 原始长历史
- 未经筛选的大段工具输出

而不应该先牺牲：

- 当前目标
- 当前约束
- 当前状态摘要

> [!warning] 误区
> 很多系统在 token 紧张时最先压缩的恰恰是目标和状态，这几乎等于让模型在预算紧张时更盲。

## 9. 典型装配反模式

理解装配设计，最直接的方式之一就是识别反模式。

### 9.1 全量追加型

新信息一来就直接塞进上下文，不做筛选、不做分层、不做摘要。

结果通常是：

- 越跑越长
- 越跑越乱
- 越跑越贵

### 9.2 最近轮迷信型

机械相信“最近几轮一定最相关”。

这会导致系统忽略更早但更关键的目标、约束和中间结论。

### 9.3 原文崇拜型

认为摘要不可靠，所以一切都应该原文保留。

结果是：

- 信息量大幅膨胀
- 当前决策密度降低

### 9.4 单模板通吃型

无论任务开始、中途还是收束，都使用完全相同的装配模板。

这会让系统失去阶段感，进而让上下文总是在“差不多但不完全对”的状态里运行。

### 9.5 检索即可信型

只要检索到了内容，就默认它值得进入当前轮。

这会把很多召回噪音误当成可用证据。

## 10. 一个可复用的装配检查框架

为了以后调试方便，你可以用下面四个问题快速检查某一轮装配质量。

### 10.1 当前轮有没有把目标和边界放在最前面

如果没有，模型往往会先被材料牵着走。

### 10.2 当前轮有没有显式状态摘要

如果没有，模型很容易在长任务里迷失推进位置。

### 10.3 当前轮带入的信息，是不是都直接服务于这一步决策

如果不是，说明噪音已经开始积累。

### 10.4 当前轮预算花在了最高价值的信息上吗

如果大量 token 被原始历史和低信号材料占用，说明装配已经开始失衡。

> [!tip] 使用建议
> 调试某个具体失败案例时，最好不要只看最终 Prompt。要复盘“这一轮为什么装了这些信息，而不是别的信息”。

## 11. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 上下文装配是运行时选择问题，不是静态拼接问题
- 结构设计和装配设计是两层不同问题
- 任务开始、中途推进和收束阶段，需要不同的装配重点
- 追加、裁剪、摘要、重写是四种基本运行时操作
- 不同来源的信息必须经过路由，而不是默认进入
- 预算管理本质上是优先级管理

如果第 4 章回答的是“信息应该怎么长”，这一章回答的就是：**长得正确的信息，在每一轮里该怎样被带入，怎样保持密度，怎样避免失真。**

## 一页总结

- 装配的本质是运行时选择，而不是机械拼接。
- 起始轮、中途轮、收束轮需要不同的装配重点。
- 追加、裁剪、摘要、重写是四个基础操作。
- 历史、检索、工具结果都应经过路由，不应默认全进。
- 预算管理本质上是优先级管理。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 系统，尝试回答下面的问题：
> 1. 它当前轮装配时，哪些信息是默认进入的？
> 2. 它会不会机械地把最近历史或工具结果全带进来？
> 3. 它有没有显式的裁剪、摘要和重写逻辑？
> 4. 如果要提升稳定性，你会先改它的装配规则，还是改它的结构设计？
