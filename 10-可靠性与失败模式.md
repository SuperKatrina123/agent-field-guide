# 10 可靠性与失败模式

> [!note] 课程说明
> **学习目标**：把 Agent 的“经常不稳定”从模糊抱怨，转化为一套可以分类、定位、复盘的失败模型。  
> **前置知识**：建议先读完 [[05-上下文装配与路由]]、[[08-规划、分解与执行控制]]。  
> **预计时间**：核心阅读 `55-75 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`Agent 失败通常有哪些类型`、`为什么错误会级联放大`、`有哪些常见防御性设计`、`如何做结构化复盘`。

---

> [!question] 带着问题阅读
> 为什么 Agent 的很多错误看起来都很像“模型不够聪明”，但真到复盘时，你会发现有的错误其实来自上下文，有的来自工具，有的来自规划，有的来自控制权设计？如果失败来源不同，为什么团队总喜欢用同一种方式去修？

## 1. 为什么必须建立失败分类法

Agent 最难调的地方之一，不是它一定比普通系统更容易出错，而是它的错误看起来经常“像一回事”。

典型感受包括：

- 它怎么又做错了
- 它怎么又跑偏了
- 它怎么又忘了
- 它怎么又乱调工具了

这些抱怨都是真的，但如果一直停留在这种层面，系统几乎不可能稳定进化。

因为不同失败，对应的修法完全不同：

- 上下文污染不能靠换工具修
- 工具误用不能靠加记忆修
- 目标偏移不能靠加更多示例修

所以，这一章最核心的任务是：**把“Agent 不稳定”拆成可区分的失败类型。**

> [!abstract] 定义
> 本章所说的“可靠性”，不是指 Agent 永远不出错，而是指系统在常见任务、常见异常和常见边界条件下，能够以可理解、可定位、可恢复的方式运行。

## 2. 常见失败不只是一种“笨”

Agent 的失败，大致可以从五个层面理解。

### 2.1 目标层失败

这类失败通常表现为：

- 目标理解错了
- 完成标准抓错了
- 把局部目标当成全局目标

### 2.2 上下文层失败

这类失败通常表现为：

- 忽略关键约束
- 带入无关历史
- 状态断裂
- 长任务漂移

### 2.3 工具层失败

这类失败通常表现为：

- 选错工具
- 错读工具结果
- 失败后继续错误推进

### 2.4 控制层失败

这类失败通常表现为：

- 明明该停却不停
- 明明该补问却直接猜
- 明明该重规划却盲目继续

### 2.5 协作层失败

在 Workflow 或多 Agent 系统里，还会出现：

- 分工失真
- 汇总失真
- 状态不同步

## 3. 几类最典型的失败模式

### 3.1 幻觉型失败

幻觉不是只发生在知识问答里。

在 Agent 系统里，幻觉还包括：

- 编造不存在的状态
- 假装工具调用成功
- 把未确认信息当事实

### 3.2 漏步骤型失败

系统跳过了本应执行的关键步骤。

常见原因包括：

- 计划不完整
- 当前状态摘要缺失
- 完成标准太模糊

### 3.3 误用工具型失败

系统使用了不该用的工具，或者在不该行动时行动。

常见原因包括：

- 工具边界不清
- 决策条件不清
- 返回值结构太差

### 3.4 目标偏移型失败

系统一开始目标理解是对的，但随着任务推进逐渐偏离。

这类失败经常和上下文漂移、阶段切换不清有关。

### 3.5 级联型失败

最麻烦的一类失败不是单点错误，而是：

- 前一轮小错
- 下一轮被当真
- 再下一轮继续放大

最后系统看起来像“整体都坏了”。

## 4. 为什么错误会级联放大

Agent 系统的核心结构决定了它的很多错误天然具有传播性。

因为：

- 当前轮结果会变成下一轮输入
- 当前轮状态会影响下一轮决策
- 当前轮错误工具结果可能被当成未来事实

```mermaid
flowchart LR
    A["局部错误"] --> B["写入状态或上下文"]
    B --> C["影响下一轮决策"]
    C --> D["触发更大错误动作"]
```

所以，很多 Agent 失败的危险并不在“它偶尔错一次”，而在“它会把这次错继续带着往下跑”。

## 5. 上下文漂移与目标偏移如何区分

这两个词很容易混。

### 5.1 上下文漂移

更强调的是：

- 当前轮带给模型的信息逐渐偏离原始任务需要

例如：

- 旧历史不断累积
- 局部子问题压过全局目标
- 暂时性材料占据主要注意力

### 5.2 目标偏移

更强调的是：

- 系统整体正在围绕错误目标推进

例如：

- 把“收集信息”误当成最终目标
- 把“修一个局部问题”当成整个任务完成

这两者常常一起出现，但修法不同。

## 6. 计划失真与执行失控如何区分

### 6.1 计划失真

指一开始的任务分解、步骤设计、终止条件本身就有问题。

### 6.2 执行失控

指原计划本来还行，但系统在执行中没有守住控制边界。

例如：

- 没按计划检查关键状态
- 遇到失败仍然盲进
- 该确认时没确认

## 7. 为什么“模型不够强”经常是最差诊断

“模型不够强”有时确实是原因之一，但它通常不是最先该给出的诊断。

因为这是一个太宽的解释。

它会掩盖真正可行动的问题：

- 当前轮有没有带对目标
- 状态有没有维护
- 工具结果有没有结构化
- 执行条件有没有设计

如果这些问题不先排，换更强模型也常常只是把错误推迟一点，而不是消除。

> [!warning] 误区
> 把多数失败都解释成模型能力不足，往往意味着系统设计层还没有建立基本诊断能力。

## 8. 防御性设计到底在防什么

所谓防御性设计，不是给系统加很多限制，而是防止局部错误轻易扩散成系统性错误。

### 8.1 防漂移

常见办法包括：

- 明确带入当前目标
- 阶段切换时重申完成标准
- 让状态字段持续可见

### 8.2 防误用工具

常见办法包括：

- 工具职责清晰化
- 失败结构化
- 高风险动作确认

### 8.3 防级联

常见办法包括：

- 关键检查点
- 状态复核
- 局部失败不立即写成长期事实

### 8.4 防无限推进

常见办法包括：

- 明确终止条件
- 预算上限
- 重试上限

## 9. 失败复盘应该怎么做

很多团队复盘 Agent 失败时，只会记录：

- 用户问题是什么
- 错误结果是什么

这远远不够。

更有效的复盘，至少要回答：

- 哪一轮开始偏
- 当时系统看到了什么
- 当时系统用了什么工具或动作
- 状态是如何更新的
- 错误是如何向后传播的

### 9.1 一个最小复盘结构

1. 任务目标
2. 失败表现
3. 首个关键偏差点
4. 失败类型归类
5. 传播路径
6. 可阻断点

### 9.2 复盘的重点不是归责，而是找可阻断点

如果复盘最后只能落在“模型表现不稳定”，那基本没有沉淀价值。

真正有价值的问题是：

- 哪一步本来可以拦住它
- 哪个信号本来应该被系统看见

## 10. 典型反模式：错误分类混乱

### 10.1 把所有错误都叫幻觉

这会直接失去诊断能力。

### 10.2 把所有错误都修在 Prompt 上

这会让系统不断变长，但不一定更稳。

### 10.3 看到一次失败就大改架构

很多问题只是局部控制或上下文问题，不需要升级成多 Agent 或引入更复杂流程。

## 11. 一个可复用的失败分析框架

### 11.1 先问失败属于哪一层

- 目标
- 上下文
- 工具
- 控制
- 协作

### 11.2 再问首个偏差点在哪一轮

不要只看最终错误结果，要找第一处偏差。

### 11.3 再问它如何传播

很多失败不是第一次错最严重，而是第一次错之后没人拦。

### 11.4 最后问系统本可在哪阻断

这是最重要的问题，因为它直接指向改进点。

> [!tip] 使用建议
> 真正成熟的可靠性建设，不是追求“从不失败”，而是让失败更早被发现、更容易被阻断、更容易被复盘。

## 12. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- Agent 的失败不是单一类型，而是多层问题的混合表现
- 很多严重失败都具有级联特征
- “模型不够强”通常不是最有效的第一诊断
- 防御性设计的核心，是阻断局部错误向系统性错误扩散
- 高质量复盘的关键，是找到首个偏差点和可阻断点

## 一页总结

- Agent 失败不是单一类型，而是目标、上下文、工具、控制、协作等多层问题。
- 最危险的错误不是单次错，而是错误被继续带入后续轮次。
- “模型不够强”通常不是最先该给出的诊断。
- 防御性设计的核心是阻断错误传播，而不是事后解释。
- 高质量复盘的重点，是找首个偏差点和可阻断点。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 失败案例，尝试回答下面的问题：
> 1. 它首先失败在目标、上下文、工具、控制还是协作层？
> 2. 第一处明显偏差发生在哪一轮？
> 3. 错误是如何被带到后续轮次里的？
> 4. 如果重做一次系统设计，你会优先放哪个阻断点？
