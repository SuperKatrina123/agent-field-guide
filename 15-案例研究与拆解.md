# 15 案例研究与拆解

> [!note] 课程说明
> **学习目标**：把前面的结构方法真正落到案例分析中，建立“如何拆一个 Agent 系统”的统一视角。  
> **前置知识**：建议先完成主线阅读，至少读到 [[10-可靠性与失败模式]]。  
> **预计时间**：核心阅读 `45-60 分钟`，案例练习 `30-40 分钟`。  
> **本章任务**：回答三个问题，`案例应该怎么看`、`拆解时要抓哪些维度`、`失败案例为什么比成功案例更有价值`。

---

> [!question] 带着问题阅读
> 为什么很多人看了很多 Agent 案例，最后却只记住“这个产品挺厉害”“这个系统好像能做很多事”，却仍然不会自己设计？案例真正应该看的是结果，还是结构？

## 1. 案例的价值，不是看热闹

案例研究最大的价值，不是给人带来“这个系统好先进”的感觉，而是把抽象原则压回真实系统。

如果前面章节给的是设计语言，那案例页要做的，就是训练你把这套语言用起来。

所以，读案例时最该看的不是：

- 名字有多大
- 演示有多炫

而是：

- 目标是什么
- 上下文怎么组织
- 状态怎么维护
- 工具怎么接
- 失败怎么暴露

## 2. 一个案例应该怎么拆

最简单也最有效的方式，是沿着前面整套课程的主线往下拆。

### 2.1 先看目标

这个系统到底在完成什么？

如果目标本身说不清，后面的复杂结构很可能只是热闹。

### 2.2 再看控制

系统的下一步动作是谁决定的？

是人、流程、还是运行时控制环？

### 2.3 再看上下文

模型每轮真正看见了什么？

### 2.4 再看工具与环境

它接了哪些能力？这些能力风险在哪里？

### 2.5 最后看评测与失败

系统是怎么知道自己做得好不好的？

## 3. 成功案例最容易误导人的地方

成功案例经常有一个天然缺陷：

- 你能看到结果
- 但看不到失败成本

这会导致很多误读，例如：

- 觉得多 Agent 一定有效
- 觉得工具越多越强
- 觉得长上下文一定带来更强表现

所以看成功案例时，必须反向追问：

- 它背后的控制成本是什么
- 哪些结构只是这个场景下成立
- 哪些方法一换场景就会失效

## 4. 为什么失败案例更有价值

失败案例的价值在于它会暴露系统的真实边界。

你可以更清楚地看到：

- 第一处偏差在哪里
- 错误如何传播
- 哪一步本来可以拦住它

相比成功案例，失败案例更能训练：

- 诊断能力
- 结构敏感度
- 风险判断

## 5. 单 Agent 案例应该重点看什么

更适合重点看：

- 单一控制环是否足够
- 状态是否清楚
- 上下文是否稳定
- 工具是否最小必要

## 6. 工具型 Agent 案例应该重点看什么

更适合重点看：

- 工具边界是否清楚
- 工具结果是否结构化回流
- 高风险动作是否有确认机制

## 7. 多 Agent 案例应该重点看什么

更适合重点看：

- 分工是否真实
- 共享上下文是否合理
- 汇总是否失真
- 协作成本是否值得

## 8. 从需求到设计的完整案例应该怎么读

最有训练价值的，不是“最终架构图”，而是：

- 为什么选这条路径
- 放弃了哪些路径
- 哪些约束主导了设计

因为这才是真实设计过程。

## 9. 一个可复用的案例拆解框架

### 9.1 目标层

它到底要解决什么问题？

### 9.2 架构层

它为什么是单 Agent、Workflow，还是多 Agent？

### 9.3 上下文层

运行时信息系统是怎样组织的？

### 9.4 工具层

它如何接环境，又如何控风险？

### 9.5 失败层

它最容易先坏在哪？

## 10. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 案例研究不是看结果炫不炫，而是看结构是否成立
- 成功案例要反向看成本与边界
- 失败案例更适合训练诊断能力
- 统一拆解框架，比记住很多零散案例更有价值

## 一页总结

- 案例研究不是看结果炫不炫，而是看结构是否成立。
- 成功案例要反向看成本与边界，失败案例更适合训练诊断力。
- 目标、架构、上下文、工具、失败层是最实用的拆解主线。
- 统一拆解框架，比积累零散案例更有用。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 产品或内部系统，尝试按下面五层拆它：
> 1. 目标层
> 2. 架构层
> 3. 上下文层
> 4. 工具层
> 5. 失败层
>
> 然后回答：这个系统真正最强的设计点是什么，最脆弱的点又是什么？

## 使用建议
- 先建立理论框架，再读案例
- 读案例时优先关注：目标、上下文、状态、工具、评测五件事
