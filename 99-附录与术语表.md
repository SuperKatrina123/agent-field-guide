> [!note] 本页定位
> 这一页作为知识库的收束区，负责沉淀全书通用术语、关键概念对照、常见缩写、推荐资料与维护规范。

## 如何使用这一页

- 遇到概念混淆时，先看术语表和对照表
- 做新项目时，可把这一页当成团队统一语言参考

## 核心术语表

### Agent

在目标约束下，基于当前状态和可见上下文，对环境做出动作，并根据反馈更新下一步判断的运行时系统。

### Context Engineering

对模型每一轮决策前可见信息的来源、结构、装配、更新与约束进行系统化设计。

### Workflow

把多个已知步骤组织成稳定、可重复执行流程的结构，更强调预先定义的顺序和分支，而不是运行时自主决策。

### Prompt

面向当前一轮输入输出的表达优化手段，重点在任务表达、约束、格式和示例。

### State

系统对当前任务推进位置、已知事实、待确认问题和最近动作结果的显式表达。

### History

此前发生过的事件流或对话流，不等于当前轮应当直接使用的状态。

### Memory

超出当前轮或当前会话范围、仍值得在未来被系统保留和再次利用的信息。

### Retrieval

在当前任务需要时，从外部知识或存储中找回相关信息的机制。

### Tool Calling

把模型内部决策转化为对环境的可执行动作，并将结果回流到后续决策的过程。

### ReAct

一种边想边做的执行模式，让系统在推理和行动之间交替推进。

### Plan-and-Execute

一种先明确计划，再沿计划执行，并按需修正的执行模式。

### Human-in-the-loop

把部分关键控制权保留给人类的设计方式，常见于高风险动作、边界模糊动作和关键审核点。

### Guardrail

用于限制错误传播、约束高风险行为、阻断越界动作的规则、机制或结构设计。

### Orchestrator

在 Workflow 或多 Agent 中承担总控、任务分配和结果汇总职责的控制节点。

## 关键概念对照表

| 概念 A | 概念 B | 关键区别 |
|---|---|---|
| Prompt | Context Engineering | 前者优化当前表达，后者设计运行时信息系统 |
| Workflow | Agent | 前者强调预定义流程，后者强调运行时决策 |
| State | History | 前者服务当前控制，后者记录过去发生的事 |
| Memory | Retrieval | 前者回答“该保留什么”，后者回答“现在如何找回来” |
| Tool | Action | Tool 是动作接口，Action 是系统对环境做的事 |
| Copilot | Agent | 前者由人主控，后者拥有部分运行时控制权 |
| 评测 | 观测 | 评测回答“好不好”，观测回答“当时发生了什么” |
| 安全 | 治理 | 安全偏边界控制，治理偏组织规则、审计和责任机制 |

## 知识库维护规范

### 章节结构

主线章节默认采用以下收束结构：

- `本章应当留下的认知结论`
- `本章结构图`
- `一页总结`
- `思考练习`
- `延伸阅读`

### 延伸阅读格式

统一保留：

- `必读`
- `延伸`

不再额外加“外链资料”等中间标题。

### 新增页面时的优先原则

- 先判断它应属于主线、案例、模板还是附录
- 先补链接，再补扩写
- 尽量沿用已有术语，不重复造词

### 维护时最重要的问题

- 这个概念是否已经在别处被定义过
- 这个页面是否真的新增了结构价值
- 这个链接是否仍然稳定有效
