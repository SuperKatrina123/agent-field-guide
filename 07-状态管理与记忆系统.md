# 07 状态管理与记忆系统

> [!note] 课程说明
> **学习目标**：把 `状态`、`历史`、`记忆`、`检索` 四个经常被混用的概念拆开，并建立一套可操作的设计口径。  
> **前置知识**：建议先读完 [[03-上下文工程总论]]、[[04-上下文结构设计]]、[[05-上下文装配与路由]]。  
> **预计时间**：核心阅读 `55-75 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`什么是状态`、`什么才算记忆`、`什么时候该写记忆`、`记忆系统与检索系统的边界在哪里`。

---

> [!question] 带着问题阅读
> 为什么很多 Agent 一旦开始“记住更多”，并没有变得更稳定，反而更容易把旧事当新事、把偏好当规则、把局部经历当长期事实？问题到底出在模型不会记，还是系统根本没分清什么该被记住？

## 1. 为什么这一章必须先拆概念

在很多团队里，只要系统开始遗忘，就很快会出现一个默认解决方案：

- 上记忆
- 上向量库
- 上长期存储

这种反应看起来很自然，但往往太快了。

因为系统“忘记”一件事，可能有完全不同的原因：

- 当前任务状态没有被显式表达
- 历史没有被正确带入当前轮
- 真正需要的是外部检索，不是长期记忆
- 某条局部信息根本不应该被永久保存

如果这些情况不先拆开，“上记忆”很容易变成一种万能补丁，而不是设计判断。

> [!abstract] 定义
> 本章所说的“状态管理”，是指系统对当前任务推进过程中的动态信息进行显式表达、更新和引用；“记忆系统”则是指系统对超出当前轮、当前会话范围的信息进行可持续保留与再利用的机制。

## 2. 状态、历史、记忆、检索不是一回事

### 2.1 状态

状态回答的是：

- 现在进行到哪
- 已经知道什么
- 还缺什么
- 下一步决策应该围绕什么

状态是当前任务控制环的一部分。

### 2.2 历史

历史回答的是：

- 之前发生过什么

它更接近事件流，而不是系统当前判断。

历史很多，但不是所有历史都应当直接进入当前轮。

### 2.3 记忆

记忆回答的是：

- 哪些过去的信息值得在未来继续被系统使用

记忆的关键不是“存过”，而是“值得被长期保留并在以后继续有用”。

### 2.4 检索

检索回答的是：

- 当系统当前需要外部信息时，应该从哪里把它找回来

检索可以面向知识库，也可以面向历史存储，但它本身不是记忆。

> [!tip] 原则
> 不是所有“以后可能有用”的信息都应该进记忆系统；也不是所有“之前发生过”的信息都应该被当作当前状态。

## 3. 为什么状态必须优先于记忆

很多系统的问题，根本不是“没有记忆”，而是“没有状态”。

例如：

- 当前阶段是什么
- 上一轮做了什么
- 哪个问题已经确认
- 哪个约束仍然有效

如果这些事情没有被显式表示，系统就会出现一种典型假象：

- 看起来像在多轮推进
- 实际上每轮都在半重新开始

这时你如果直接上长期记忆，并不能解决根本问题。因为系统连当前任务都没管理清楚，就更不可能稳定使用长期信息。

## 4. 什么信息更适合当状态

状态更适合保存的是：

- 当前阶段
- 当前待办
- 已确认事实
- 待确认问题
- 最近一次关键动作结果

这些信息有几个共同点：

- 与当前任务推进直接相关
- 会频繁变化
- 生命周期通常不长

它们更像控制系统的当前面板，而不是长期知识库。

## 5. 什么信息更适合当记忆

不是所有过去信息都值得被保存为记忆。

更适合进入记忆系统的，通常是那些：

- 跨任务仍然有价值
- 在未来高概率会被再次使用
- 不是当前轮局部偶然产物

例如：

- 用户长期偏好
- 经过多次确认的稳定事实
- 某类任务的持久约束
- 反复出现的上下文背景

### 5.1 一个很重要的判断标准

问自己：

> 如果这个任务结束了，这条信息在未来仍然值得被系统主动想起吗？

如果答案是否定的，那它大概率不该进入长期记忆。

## 6. 短期记忆、中期记忆、长期记忆如何理解

这三种叫法的价值，不在于分类漂亮，而在于帮助你按生命周期设计信息。

### 6.1 短期记忆

更接近当前会话内、当前任务内的持续信息。

它通常包括：

- 当前轮附近的重要上下文
- 当前任务阶段摘要
- 本次会话内确认过的事实

很多系统里的短期记忆，本质上和状态管理高度重合。

### 6.2 中期记忆

中期记忆可以理解为跨多个任务、但又不是永久有效的信息。

例如：

- 最近一段时间内的项目背景
- 某个持续中的主题上下文
- 一段时期内仍然有效的偏好或限制

它的特点是：

- 比会话长
- 比真正长期知识短
- 需要过期和治理

### 6.3 长期记忆

长期记忆更适合保存高稳定性、高复用价值的信息。

例如：

- 用户稳定偏好
- 长期角色设置
- 经过反复确认的重要事实

长期记忆的核心不是量大，而是质量高、污染低、可持续使用。

## 7. Episodic、Semantic、Preference 记忆怎么理解

除了按生命周期，也可以按语义类型理解记忆。

### 7.1 Episodic memory

Episodic 更接近“经历过什么”。

它记录的是事件性信息，例如：

- 某次任务做过哪些决策
- 某个问题曾经怎样被处理

它的价值通常在于复盘、连续性和经验借用。

### 7.2 Semantic memory

Semantic 更接近“知道什么事实”。

它保存的是较稳定的知识性内容，例如：

- 某个系统配置
- 某类长期事实

### 7.3 Preference memory

Preference 更接近“偏好什么”。

它通常包括：

- 输出偏好
- 协作偏好
- 交互习惯

Preference 的价值很大，但也最容易被误写。因为一次临时要求，不一定等于稳定偏好。

## 8. Memory write：什么时候该写，什么时候不该写

真正难的往往不是怎么读，而是什么时候写。

### 8.1 该写的情况

更适合写入记忆的情况通常包括：

- 信息经过明确确认
- 信息跨任务有高复用价值
- 信息稳定性较高
- 信息可以被清晰归类

### 8.2 不该写的情况

很多信息不该进入长期记忆，例如：

- 当前任务中的临时假设
- 未确认事实
- 一次性局部上下文
- 高噪音工具输出
- 情绪性表达

### 8.3 写错比不写更危险

很多团队低估了“错误记忆”的代价。

因为错误记忆一旦被写入，未来每次被带回都会反复污染系统。

> [!warning] 误区
> 对记忆系统来说，写入错误信息的代价，通常比少写一条信息更高。

## 9. Memory read：什么时候该读

不是只要有记忆，就应该每轮都读。

更合理的问题是：

- 当前任务真的需要这类记忆吗
- 当前决策真的依赖这类信息吗
- 读回来的记忆会不会压过当前上下文

### 9.1 默认全读是反模式

如果系统每轮都把大量历史记忆带回当前轮，通常会导致：

- 噪音累积
- 优先级失衡
- 当前问题被旧问题干扰

### 9.2 读记忆本质上也是路由

记忆系统不是“存了就赢”，而是：

- 该读什么
- 什么时候读
- 以什么摘要形态读

这些问题本质上仍然属于上下文装配与路由。

## 10. 记忆系统和检索系统的边界

这两个系统经常被混为一谈。

更清楚的区分方式是：

- 记忆系统回答“哪些信息值得持续保留”
- 检索系统回答“当前需要时，如何把外部信息找回来”

### 10.1 检索不等于记忆

你可以对一个知识库做检索，但这并不代表系统“记住了”那个知识。

### 10.2 记忆也不等于检索

你可以把用户偏好写进长期记忆，但这不意味着它必须通过向量检索实现。

从系统角度看，二者可以配合，但职责不同。

## 11. 记忆污染与过时治理

记忆系统如果没有治理，很快就会变成污染源。

### 11.1 常见污染来源

- 未确认信息被写入
- 临时偏好被当成长期偏好
- 局部经历被误当成稳定规律
- 旧事实没有过期

### 11.2 过时问题尤其严重

很多错误记忆不是“从来没对过”，而是“曾经对过，但现在已经不对了”。

如果系统没有过期机制、复核机制或替换机制，记忆系统会从资产变成负债。

### 11.3 治理思路

至少要考虑：

- 写入门槛
- 来源可信度
- 更新时间
- 可替换性
- 过期判断

## 12. 一个可复用的记忆设计检查框架

### 12.1 这条信息真的是跨任务资产吗

如果不是，就别急着写入长期记忆。

### 12.2 它是状态、历史还是记忆

先分层，再决定存储位置。

### 12.3 它未来以什么方式被读回

如果你根本不知道未来如何用它，现在写进去多半只是堆积。

### 12.4 它会不会污染未来判断

这个问题在写入时就该问，而不是等系统变坏以后再补救。

> [!tip] 使用建议
> 设计记忆系统时，先做“少而准”的写入策略，比一开始追求“什么都记住”更重要。

## 13. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 状态、历史、记忆、检索是四个不同层级的问题
- 状态优先于记忆，当前任务没管理好时，上长期记忆通常只是补丁
- 记忆设计的难点主要在写入，不在存储
- 读记忆本质上仍然是上下文装配问题
- 记忆系统如果没有治理，迟早会反向污染 Agent

## 一页总结

- 状态、历史、记忆、检索是四个不同层级的问题。
- 当前任务没管理好时，上长期记忆通常只是补丁。
- 记忆设计最难的是写入门槛，不是存储本身。
- 读记忆仍然要经过装配与路由判断。
- 记忆系统如果没有过期和替换机制，会从资产变成污染源。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 系统，尝试回答下面的问题：
> 1. 它当前有哪些信息其实是状态，不是记忆？
> 2. 它保存的哪些信息更像历史，而不是可复用记忆？
> 3. 它有没有错误地把一次性要求当成长期偏好？
> 4. 如果你只能优化一项，你会先改写入策略、读取策略，还是过期治理？
