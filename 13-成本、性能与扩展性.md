# 13 成本、性能与扩展性

> [!note] 课程说明
> **学习目标**：把 Agent 的“能跑”进一步推进到“能持续跑、能规模化跑、能在预算内跑”。  
> **前置知识**：建议先读完 [[05-上下文装配与路由]]、[[09-Workflow与多Agent协作]]。  
> **预计时间**：核心阅读 `50-70 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`成本主要耗在哪`、`延迟为什么会堆高`、`什么时候该做压缩和缓存`、`系统如何在规模化时不失控`。

---

> [!question] 带着问题阅读
> 为什么很多 Agent Demo 看起来都能跑，但一到真实使用频率、真实上下文长度、真实工具链复杂度下，成本和延迟就立刻失控？如果能力已经做出来了，为什么离“能长期用”还差这么远？

## 1. 成本与性能不是后期优化，而是设计变量

很多项目会先追求“先跑通”，等功能出来再考虑成本和性能。

这在原型阶段没问题，但如果系统目标是长期运行或规模化使用，成本与性能就不能只当后期优化项。

因为它们会反过来影响：

- 工具策略
- 上下文策略
- 模型选择
- 架构选择

> [!abstract] 定义
> 本章所说的“成本”，不只是 token 账单，还包括工具调用、观测、协作、失败重试等总系统代价；“性能”也不只是响应速度，还包括吞吐、稳定性和在复杂任务下的可持续运行能力。

## 2. 成本主要耗在哪里

很多团队一开始只盯着模型 token 费用，但真实系统的成本通常是复合的。

### 2.1 模型调用成本

这是最直观的一层，包括：

- 输入 token
- 输出 token
- 多轮重复调用

### 2.2 工具调用成本

很多 Agent 不是“模型太贵”，而是：

- 工具太多
- 调用太频
- 失败后重试太多

### 2.3 协作与控制成本

一旦进入 Workflow 或多 Agent，额外成本会来自：

- 中间调度
- 汇总
- 路由
- 额外评审环节

### 2.4 调试与观测成本

复杂系统要想可调试，往往还要承担：

- trace 成本
- 日志成本
- 评测成本

## 3. 延迟是怎么堆出来的

很多 Agent 响应慢，不是单个模型慢，而是链路整体拖长。

### 3.1 多轮链路天然放大延迟

每多一轮：

- 就多一次模型调用
- 多一次上下文装配
- 多一次工具等待

### 3.2 工具本身可能比模型更慢

例如：

- 网页操作
- 检索系统
- 外部 API

这些常常才是长尾延迟的来源。

### 3.3 过重控制模式会堆叠延迟

如果一个系统同时有：

- 计划
- 反思
- 评审
- 汇总

那延迟往往不是线性增加，而是链路性堆高。

## 4. 为什么上下文策略直接影响成本

上下文是成本与质量之间最典型的权衡点。

### 4.1 上下文越长，成本不只是线性问题

更长上下文通常意味着：

- 输入 token 增加
- 响应更慢
- 注意力更稀释

### 4.2 低质量上下文会带来二次成本

更危险的是：

- 上下文太乱导致判断失真
- 判断失真导致多调工具
- 多调工具导致更多轮次和更高成本

所以，坏上下文会放大坏成本。

## 5. 模型选择其实是在做成本-效果权衡

很多系统默认追求“最强模型”，但从工程角度看，更好的问题是：

- 当前任务真的需要多强模型
- 哪些环节需要强模型
- 哪些环节可以用更轻模型

### 5.1 全链路统一最强模型的问题

会带来：

- 不必要的费用
- 更高延迟
- 资源利用低效

### 5.2 分层模型策略更实用

很多系统更合理的做法是：

- 高难决策点用更强模型
- 低风险格式化或路由点用更轻模型

## 6. 缓存、压缩、裁剪为什么重要

这三件事本质上都在做一件事：

> 让系统不要为重复信息和低价值信息反复付费。

### 6.1 缓存

适合：

- 稳定提示前缀
- 稳定知识片段
- 可复用中间结果

### 6.2 压缩

适合：

- 长历史
- 大体量工具结果
- 已完成阶段的中间材料

### 6.3 裁剪

适合：

- 低相关信息
- 已失效信息
- 重复信息

> [!tip] 原则
> 性能优化里最有效的一类动作，通常不是“让模型更快”，而是“不要反复让模型看低价值内容”。

## 7. 并发与吞吐设计在 Agent 里为什么更难

普通接口的并发问题，通常主要是资源问题。

Agent 的并发问题还会额外叠加：

- 状态一致性
- 工具竞争
- 上下文隔离
- 日志与审计量暴增

### 7.1 并发不是简单复制调用

如果多个任务共用：

- 相同工具资源
- 相同上下文空间
- 相同状态存储

系统很容易相互污染或相互拖慢。

### 7.2 吞吐设计需要先保证隔离

如果隔离做不好，后面再谈吞吐通常只是在放大故障传播。

## 8. 为什么多 Agent 常常意味着更高成本

很多人以为多 Agent 能把复杂任务拆开，所以应该更高效。

现实里它经常更贵，原因包括：

- 更多模型调用
- 更多中间通信
- 更多汇总
- 更多可观测性负担

所以，多 Agent 只有在它确实显著提高结果质量、稳定性或职责隔离时，才值得这些额外成本。

## 9. 扩展性不只是“能加机器”

Agent 系统的扩展性不只看吞吐。

它还要看：

- 上下文复杂度是否会持续膨胀
- 工具数量增加后是否还能控
- 状态系统能否保持清晰
- 评测与治理成本是否同步增长

如果这些都在失控，系统即使短期吞吐还能顶住，也很难真正扩展。

## 10. 常见成本与性能反模式

### 10.1 上下文全保留型

所有历史都带、所有结果都留。

结果是：

- 账单高
- 延迟高
- 质量未必高

### 10.2 最强模型通吃型

不分环节地一律使用最强模型。

### 10.3 失败无限重试型

局部错误不做分类，直接重复调用，快速放大成本。

### 10.4 多 Agent 先上再说型

还没证明收益，就先承担协作成本。

## 11. 一个可复用的成本-性能检查框架

### 11.1 当前最大成本来源是什么

先分清是模型、工具、控制还是重试成本。

### 11.2 当前最大延迟来源是什么

不要默认怪模型，工具和控制链也常常是主因。

### 11.3 哪些信息在被重复消费

缓存和压缩通常先从这里见效。

### 11.4 当前架构复杂度是否值得

如果复杂度带来的收益不明显，就该考虑收缩。

> [!tip] 使用建议
> 性能优化最忌讳“哪里慢就哪里补丁”。先画出完整链路，再决定该压缩上下文、换模型、减轮次还是减工具。

## 12. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 成本和性能会反向塑造 Agent 架构，不只是后期优化项
- 坏上下文会通过额外轮次和错误动作放大总成本
- 模型选择本质上是成本-效果权衡
- 缓存、压缩、裁剪通常是最有效的一类优化手段
- 多 Agent 只有在收益足够大时才值得它的协作成本

## 一页总结

- 成本与性能会反向塑造 Agent 架构，不只是后期优化项。
- 坏上下文会通过额外轮次和错误动作放大总成本。
- 模型选择本质上是在做成本-效果权衡。
- 缓存、压缩、裁剪往往比单纯换模型更有效。
- 多 Agent 只有在收益足以覆盖协作成本时才值得。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 系统，尝试回答下面的问题：
> 1. 它当前最大的成本来源是模型、工具，还是重试与协作？
> 2. 它最大的延迟瓶颈在哪一段链路？
> 3. 哪些上下文内容其实在被重复带入？
> 4. 如果只能做一项优化，你会优先做缓存、裁剪、换模型，还是减轮次？
