# Style Guide

> [!abstract] 这份文档是什么
> Agent Field Guide 的写作与排版规范。无论你是贡献知识点、补充案例还是提交面试题，都请先读这一页。
> 目标不是把文档做得花哨，而是建立一套**可复制、可扫描、可长期维护**的知识库视觉语言。

**适用范围**：所有 `.md` 贡献文件，包括章节主文档、概念子笔记、案例笔记。  
**技术边界**：只使用 Obsidian 原生 Markdown + Callout，不依赖 CSS Snippet 或主题定制；同时保证在 GitHub 上可读。  
**关键词**：`结构化`、`工具化`、`边界清晰`、`重点明确`、`长期阅读友好`

---

## 核心设计理念

### 1. 结构化优先于装饰感
知识库首先是知识系统，不是视觉作品。页面的美感来自稳定结构、清晰层级和可预期的重点分布，而不是大量颜色、图标或组件。

### 2. 工具化优先于情绪化表达
每一段内容都应当让读者快速判断：

- 这是什么信息
- 为什么它重要
- 我该如何使用它

因此，高亮必须带有稳定语义，而不是"作者觉得重要就加粗"。

### 3. 正文优先于提示块
主阅读区必须保持连续。Callout 的作用是建立信息边界，而不是打断阅读。一个页面可以有重点，但不能整页都在高亮。

### 4. 页面必须支持快速扫描
读者打开任一章节，应该能在 10 秒内识别：

- 本页目标
- 当前阅读主线
- 哪些内容是定义
- 哪些内容是方法
- 哪些内容是误区

> [!tip] 核心判断
> 如果一页内容必须逐行细读才能知道重点，说明这页的结构设计失败了。

### 5. 每页服务整体，而不是孤立存在
页面既要能独立阅读，也要能通过内链和导航回到知识库主线。

## 设计禁忌

> [!warning] 下列问题一旦出现，会直接破坏文档的可读性与系统感

| 禁忌 | 具体表现 |
|------|---------|
| **过度设计** | 大面积彩色区块、堆叠图标表情、把页面做成"组件展板" |
| **模糊边界** | 定义和观点混在一起、正文和补充说明不分、练习和案例共用同一种表达方式 |
| **过多高亮** | 连续出现多个高亮段落、整句加粗代替结构设计、多种强调方式叠加 |
| **标题层级混乱** | 跳级使用标题、H3 当视觉装饰、一页堆太多平级模块 |
| **正文被提示块打碎** | 页面主体是一串 callout 而不是正文 |

> [!warning] 反模式判断
> 当一个页面看起来"很多重点"，实际效果通常是"没有重点"。

## 布局规范

### 页面节奏
所有文档统一采用 `导读区 → 正文区 → 收束区` 的节奏。

| 区域 | 作用 | 典型内容 |
|------|------|---------|
| **导读区** | 建立上下文 | 页面定位、学习目标、前置知识、带着问题阅读 |
| **正文区** | 知识展开 | 按 H2 / H3 展开，每个 H2 只讲一个清晰问题 |
| **收束区** | 帮助收束 | 本节小结、相关内链、下一节预告 |

### 段落与列表
- 单段优先控制在 `2–4 行`，超过 5 行优先拆段
- 一个模块内连续列表不超过 `5–7 项`
- 列表项尽量只表达一个要点；若需要展开推理，转为正文或 H3

### 分隔线与空白
- 分隔线只用于大模块切换，每页最多 `3–5` 条
- 允许留白，短段落比密集长文更适合阅读

> [!info] 布局原则
> Markdown 的"版心感"不靠固定像素宽度实现，而靠**短段落 + 稳定标题层级 + 克制的模块数量**建立。

## 标题与文字规范

### 标题职责

| 级别 | 用途 |
|------|------|
| `H1` | 页面标题，仅出现一次 |
| `H2` | 一级模块 |
| `H3` | 模块内子主题 |
| `H4` | **不使用**，需要 H4 时通常说明页面应当拆分 |

### 加粗规则

✅ 用于：结论词、关键对比词、需要被快速扫到的判断口径  
❌ 不用于：整句加粗、连续多行加粗、用加粗代替小标题

### 引用块
普通引用块只用于引述或轻量旁注，不与 Callout 共同承担同一层表达。

## Callout 语义规范

> [!abstract] 总原则
> 所有重点块必须按**信息类型**分类，不按"作者主观重要程度"分类。

课程统一使用以下 6 类：

| 类型 | 语法 | 用途 |
|------|------|------|
| 定义 | `[!abstract]` | 概念界定、术语说明、关键判断口径 |
| 原则 | `[!tip]` | 设计原则、思维框架、判断标准 |
| 方法 | `[!info]` | 步骤、操作路径、执行建议 |
| 误区 | `[!warning]` | 常见错误、反模式、易混淆点 |
| 案例 | `[!example]` | 正例、反例、场景拆解 |
| 练习 | `[!question]` | 行动任务、思考题、实践入口 |

**使用上限**：同一屏 ≤ 2 个，同一 H2 ≤ 3 个。超出时说明应回到正文结构重写。

### 示例

```md
> [!abstract] 定义
> Agent 设计中的"上下文工程"，不是单纯写 Prompt，而是对输入模型的全部有效上下文进行结构化设计。

> [!tip] 原则
> 先保证边界清晰，再追求能力丰富；边界不清的 Agent，后期很难优化。

> [!warning] 误区
> 把所有历史消息都塞进上下文，并不等于"记忆更强"，通常只会带来噪音和成本上升。
```

## 内链与标签规范

### 内链规则

**必须内链**：关键术语、单独成页的方法论、重复出现的核心概念、重要案例页  
**不强制内链**：一次性说明、临时举例、不会复用的普通名词

> [!info] 内链原则
> 首次出现且值得展开的概念使用 `[[双链]]`；同一页面内重复出现同一概念不必每次都链接。正文内链数量要克制，避免一段里出现多个双链。

### 标签规则
标签只承担归类职责，建议保留以下稳定维度：

```
主题：#agent  #context-engineering
页面类型：#chapter  #concept  #case  #interview
状态：#draft  #stable
```

## 贡献模板

### 知识点补充模板

```md
# 概念名 / 知识点名

> [!abstract] 一句话定义
> 用一句话说清这个概念是什么。

## 它解决什么问题

## 核心结构或原理

## 与相近概念的区别

## 在 Agent 设计中的位置

## 相关页面

- [[相关章节]]
- [[相关案例]]
```

### 踩坑经验模板

```md
# 坑：[简短描述]

> [!example] 场景摘要
> 在什么情境下踩了什么坑，最终怎么解决。

## 场景

## 问题表现

## 根本原因

## 解决方案

## 可迁移结论
```

### 面试题模板

```md
# [面试题题目]

**难度**：⭐ / ⭐⭐ / ⭐⭐⭐  
**考点**：上下文工程 / 记忆系统 / 工具设计 / 多 Agent / ...

## 标准答案

## 加分点

## 相关知识点

- [[相关章节]]
```

## 发布前检查清单

在提交 PR 前，请逐项确认：

- [ ] 页面一眼可见主线
- [ ] H1 / H2 / H3 职责清晰，未使用 H4
- [ ] 只使用了约定好的 6 类 Callout
- [ ] 没有连续过多高亮
- [ ] 正文未被提示块打碎
- [ ] 内链服务理解，而不是制造跳转噪音
- [ ] 使用了对应的贡献模板

> [!tip] 验收标准
> 读者打开页面时，能快速知道"这页讲什么、重点在哪、该怎么继续读"——而不是被页面样式本身分散注意力。
