# 16 模板、清单与工作台

> [!note] 课程说明
> **学习目标**：把整套知识库从“理解方法”推进到“直接产出设计文档、评测方案和复盘记录”。  
> **前置知识**：建议至少读完 [[08-规划、分解与执行控制]] 和 [[11-评测、观测与调试]]。  
> **预计时间**：核心阅读 `40-50 分钟`，模板实操 `30-60 分钟`。  
> **本章任务**：回答三个问题，`做 Agent 时该交付哪些文档`、`哪些检查清单最值钱`、`如何把方法固化成日常工作台`。

---

> [!question] 带着问题阅读
> 为什么很多团队明明已经掌握了不少 Agent 理论，但一到真正做设计文档、上线评审和复盘时，还是只能靠口头沟通和临场发挥？如果知识不能变成工作台，它到底沉淀了什么？

## 1. 为什么模板和清单很重要

前面章节讲的是判断。

但真实工程里，仅靠脑中判断远远不够。

因为一个系统真正落地时，团队需要的是：

- 能复用的设计文档结构
- 能快速检查风险的清单
- 能对比版本的评测模板
- 能复盘失败的固定框架

如果没有这些东西，知识库再完整，最后也很容易退化成“读过很多，但做的时候还是从零开始”。

## 4. 工具接入检查清单

在接入一个新工具前，先过这一轮检查。

- [ ] 这个工具的职责是否单一清楚？
- [ ] 它属于 `读`、`写` 还是 `执行` 工具？
- [ ] 模型是否容易理解它“什么时候该用，什么时候不该用”？
- [ ] 工具名是否表达动作语义，而不是内部实现细节？
- [ ] 参数是否反映决策边界，而不是暴露过多底层实现？
- [ ] 返回值是否能清楚表达成功 / 失败 / 不确定性？
- [ ] 失败信息是否结构化，而不是只给一段原始错误？
- [ ] 原始结果是否需要和决策摘要分开保存？
- [ ] 这个工具是否会对外部环境产生副作用？
- [ ] 如果有副作用，是否已定义风险级别和确认机制？
- [ ] 是否已定义权限范围，而不是默认全开放？
- [ ] 是否已明确这个工具最容易被误用的方式？

## 5. 记忆设计检查清单

在决定“把什么写进记忆”前，先过这一轮检查。

- [ ] 这条信息是 `状态`、`历史`、`记忆`，还是 `检索材料`？
- [ ] 它在当前任务结束后是否仍然有高复用价值？
- [ ] 它是经过确认的事实，还是临时推断？
- [ ] 它属于偏好、事实，还是某次局部经历？
- [ ] 未来系统会在什么场景读回它？
- [ ] 如果未来不会稳定读回，现在写入是否只是堆积？
- [ ] 这条信息是否可能过时？
- [ ] 如果会过时，是否定义了更新时间、替换机制和过期判断？
- [ ] 如果写错，这条记忆是否会反复污染未来判断？
- [ ] 当前系统真正缺的是长期记忆，还是当前任务状态管理？

## 8. 上线前审查清单

如果现在准备把一个 Agent 能力上线，先过这张表。

- [ ] 这个系统为什么必须是 Agent，而不是 Workflow / Automation / Copilot？
- [ ] 当前目标和成功标准是否清楚？
- [ ] 当前状态字段是否已经明确？
- [ ] 上下文分层是否已经定义？
- [ ] 工具集合是否是最小必要集？
- [ ] 高风险动作是否已定义确认机制？
- [ ] 权限是否最小化，而不是默认开大？
- [ ] 提示注入和上下文污染是否有基础防线？
- [ ] 是否能留下关键 trace、状态快照和工具日志？
- [ ] 是否已有最小回归集，而不是只靠演示？
- [ ] 是否知道“什么情况下必须中断，而不是继续执行”？
- [ ] 是否已经定义上线后的重点观测指标？

## 9. 怎么把这一页变成真正工作台

光有模板还不够，关键是把它变成团队动作。

一个更实用的做法是：

- 新项目立项时，先复制 `Agent 设计文档模板`
- 设计上下文前，先复制 `上下文设计模板`
- 每接一个工具，先跑 `工具接入检查清单`
- 每次版本迭代，先补 `评测方案模板`
- 每次线上异常，先走 `故障复盘模板`
- 每次准备上线，先跑 `上线前审查清单`

当这些动作稳定下来，知识库就不再只是“教材”，而真正变成“工作台”。

## 11. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 模板和清单不是官僚负担，而是把方法固化成动作
- 真正有价值的模板，应覆盖设计、评测、风险和复盘
- 知识库只有进入工作台层，才真正具备团队复用价值

## 一页总结

- 模板和清单不是官僚负担，而是把方法固化成动作。
- 设计、评测、风险、复盘都应该有对应交付物。
- 真正有价值的知识库，最后要落成工作台。
- 能直接被团队复用的，不是理念，而是稳定动作。
