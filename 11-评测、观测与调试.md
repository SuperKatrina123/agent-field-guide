# 11 评测、观测与调试

> [!note] 课程说明
> **学习目标**：把“Agent 好不好用”从主观感受，转成一套可定义、可观测、可复现、可比较的工程判断。  
> **前置知识**：建议先读完 [[10-可靠性与失败模式]]。  
> **预计时间**：核心阅读 `55-75 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`为什么 Agent 难评`、`怎样定义成功`、`系统该观测什么`、`如何从具体失败走到系统优化`。

---

> [!question] 带着问题阅读
> 为什么很多团队明明已经做了大量 Agent 迭代，却始终说不清“这一版到底有没有变好”？如果用户反馈总是“有时很好，有时很差”，系统到底该怎么被评和被调？

## 1. 为什么 Agent 比普通功能更难评测

普通功能往往有更清晰的输入输出边界：

- 给一个输入
- 看一个结果
- 判断对不对

但 Agent 系统不是这样。

它往往具有：

- 多轮过程
- 动态工具调用
- 中间状态更新
- 受上下文变化影响的行为路径

这意味着，评测不再只是“最终结果对不对”，还包括：

- 过程是不是合理
- 工具有没有用对
- 是否在正确时机停止
- 是否在错误出现时及时暴露

> [!abstract] 定义
> 本章所说的“评测”，不是泛泛地判断“像不像智能”，而是对系统在特定任务、特定约束和特定成功标准下的行为质量进行系统性验证；“观测”则是让这些行为在运行中变得可见、可追溯、可分析。

## 2. 先定义成功，否则评测没有对象

很多 Agent 项目最大的评测问题，不是没有工具，而是根本没有统一成功定义。

团队常见说法包括：

- 这版感觉更好了
- 用户大概会更喜欢
- 稳定性应该提升了

这些都不够。

更可用的评测前提是：

- 这类任务什么叫完成
- 这类任务什么叫失败
- 哪些失败可以接受
- 哪些失败绝对不能接受

### 2.1 Task success 不应该只看最终文本

对 Agent 来说，最终文本只是成功的一部分。

你往往还需要关心：

- 是否遗漏关键步骤
- 是否违反关键约束
- 是否误用了工具
- 是否过度执行

### 2.2 一个简单成功定义框架

可以先从三层想：

- 结果层：最终输出是否满足目标
- 过程层：关键过程是否合理
- 风险层：是否触发了不可接受行为

> [!tip] 原则
> 对 Agent 来说，“看起来聪明”不是评测标准；“在定义好的任务与边界内稳定完成”才是。

## 3. 评测维度应该怎么拆

一个更稳的做法，是先把评测拆成几个维度，而不是用一个模糊总分覆盖一切。

### 3.1 任务完成度

最基本的问题：

- 任务有没有完成
- 完成质量如何

### 3.2 过程合理性

Agent 的很多问题不是最终结果错，而是过程明显不合理，例如：

- 明明该补问却直接猜
- 明明该停却继续执行
- 明明不该调工具却调了

### 3.3 工具使用质量

对于工具型 Agent，至少要看：

- 工具选择是否合理
- 参数是否正确
- 结果是否正确理解

### 3.4 稳定性

不是只看一次表现，而是看：

- 同类任务是否稳定
- 输入稍有变化是否大幅波动

### 3.5 安全性与边界遵守

这类维度通常包括：

- 是否违反权限边界
- 是否发生提示注入影响
- 是否执行高风险动作

## 4. 离线评测、在线评测、人审、规则评审怎么组合

没有一种评测方式能覆盖所有问题。

### 4.1 离线评测

优点：

- 可复现
- 易比较
- 适合版本回归

缺点：

- 容易和真实环境脱节

### 4.2 在线评测

优点：

- 更接近真实场景
- 能看到真实分布下的问题

缺点：

- 噪声大
- 排查难

### 4.3 人审

优点：

- 对复杂任务和细腻质量判断更有价值

缺点：

- 成本高
- 一致性难保证

### 4.4 规则评审

适合判断：

- 是否触发高风险动作
- 是否满足明确格式约束
- 是否漏掉关键结构

> [!info] 方法
> 更成熟的做法通常不是选一种评测，而是按维度组合：离线看回归，在线看真实，人审看复杂质量，规则看硬边界。

## 5. 为什么观测比很多人想象中更重要

很多团队直到系统频繁失败后，才意识到自己其实几乎看不到 Agent 在做什么。

常见问题包括：

- 只看最终输出
- 不知道中间调了什么工具
- 不知道哪一轮开始偏
- 不知道当前状态如何变化

没有观测，调试就只能靠猜。

## 6. Trace、日志、状态快照分别服务什么

### 6.1 Trace

Trace 更适合看完整行为链。

它回答的是：

- 这次任务经历了哪些轮次
- 每轮做了哪些动作
- 哪里开始异常

### 6.2 日志

日志更适合记录关键事件和局部信号。

它回答的是：

- 某个动作是否发生
- 某次调用是否失败
- 某个工具耗时是否异常

### 6.3 状态快照

状态快照更适合看系统在关键节点“当时认为自己处于什么状态”。

它对于定位这类问题非常关键：

- 为什么明明该结束却没结束
- 为什么明明缺信息却没有补问

## 7. 评测集应该怎么设计

很多评测失败，不是执行失败，而是评测集本身太弱。

### 7.1 评测集不能只收“正常题”

如果所有样例都是最顺的路径，你测不出系统真正的问题。

更好的评测集应该覆盖：

- 正常路径
- 边界条件
- 高风险条件
- 干扰条件

### 7.2 样例要覆盖典型失败模式

如果你已经知道系统常见失败是：

- 漏步骤
- 误用工具
- 长任务漂移

那评测集就应该显式包含这些类型。

### 7.3 评测集要服务版本比较

评测集不是知识展览，而是版本比较工具。

所以它最重要的属性之一是：

- 稳定
- 可重复
- 可覆盖关键风险点

## 8. 调试 Agent 时，最怕什么

最糟糕的调试方式通常是：

- 看一两个 bad case
- 凭感觉改 Prompt
- 再看是否“感觉更好了”

这几乎注定低效。

### 8.1 更好的调试顺序

一个更稳的顺序通常是：

1. 先定位失败属于哪一层
2. 再定位首个偏差点
3. 再看当时的上下文、状态和动作
4. 最后决定改结构、装配、工具、控制还是评测本身

### 8.2 从个例走向系统问题

一个 bad case 的价值，不在于它本身，而在于它是否暴露了某类系统性风险。

如果一个问题只在一个极端样例里出现，和在 20% 常见任务里出现，处理优先级应完全不同。

## 9. 从个例调试到系统优化

系统优化不应该只是“修掉眼前这个 case”。

更重要的是追问：

- 它属于哪一类失败
- 这类失败在系统里出现频率如何
- 有没有更通用的修法

例如：

- 这不是一个 bad prompt，而是一类状态字段缺失
- 这不是一次误选工具，而是一类工具接口设计不清

## 10. 典型反模式：评测看起来做了，其实没做成

### 10.1 只看最终答案型

忽略过程质量，导致很多系统明明过程极差但“偶尔答对”仍被误判为好。

### 10.2 只看少量演示型

拿少数 showcase 代替真实评测。

### 10.3 评测维度混成一个总分型

把过程、结果、安全、成本全部揉成一个模糊分数，最后失去诊断能力。

### 10.4 没有回归集型

每次都凭主观感受判断“这一版大概更好了”，系统很难稳步迭代。

## 11. 一个可复用的评测与调试框架

### 11.1 先定义成功

没有成功定义，谈不上评测。

### 11.2 再拆评测维度

结果、过程、工具、安全、稳定性最好分开看。

### 11.3 再建立观测信号

至少要能看到：

- 当前轮上下文
- 关键状态
- 工具调用
- 终止条件

### 11.4 最后从个例归类成系统问题

不然你只是在修散点。

> [!tip] 使用建议
> 高质量的 Agent 调试，不是“看到一个错就改一句提示”，而是让错误能够被定位、分类、比较和复现。

## 12. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- Agent 的评测必须同时覆盖结果、过程和边界
- 评测前先定义成功，否则没有统一对象
- Trace、日志、状态快照分别服务不同粒度的观测
- 好的调试不是从最终答案倒推猜测，而是从首个偏差点追溯
- 从个例上升到系统问题，是 Agent 工程成熟的关键一步

## 一页总结

- Agent 评测必须同时覆盖结果、过程和边界。
- 没有成功定义，评测就没有统一对象。
- 观测系统至少要能看到当前轮上下文、关键状态、工具调用和终止条件。
- 调试的关键不是猜模型，而是找首个偏差点。
- 真正成熟的优化，是把个例问题上升成系统问题。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 系统，尝试回答下面的问题：
> 1. 它现在有没有明确的成功定义？
> 2. 它目前看的是最终结果，还是也看过程质量？
> 3. 你能不能清楚定位某个 bad case 第一次偏掉的轮次？
> 4. 如果现在只能补一项能力，你会优先补评测集、观测信号，还是调试流程？
