# 12 安全、权限与治理

> [!note] 课程说明
> **学习目标**：把 Agent 的安全问题从“担心它乱来”推进到权限、边界、确认、审计和治理的系统设计。  
> **前置知识**：建议先读完 [[06-工具使用与环境交互]]、[[11-评测、观测与调试]]。  
> **预计时间**：核心阅读 `55-75 分钟`，思考练习 `20-30 分钟`。  
> **本章任务**：回答四个问题，`提示注入与上下文污染如何防`、`为什么权限必须最小化`、`高风险动作如何控`、`企业场景下如何做审计与治理`。

---

> [!question] 带着问题阅读
> 为什么一个在实验环境里“很能干”的 Agent，一旦上线接入真实工具、真实数据和真实权限，风险就会陡增？如果能力放大了，为什么边界设计不能继续停留在“相信模型会听话”？

## 1. 安全不是附加层，而是系统边界

很多团队在前期做 Agent 时，安全常被放在后面。

原因并不奇怪：

- 先做通路
- 先看能不能跑
- 先把功能接出来

但一旦系统真正开始接环境，安全就不再是补充问题，而是结构问题。

因为这时 Agent 不只是“会说错”，它可能还会：

- 看见不该看见的信息
- 用到不该用的工具
- 执行不该执行的动作
- 在错误上下文里做出高风险判断

> [!abstract] 定义
> 本章所说的“安全”，不是泛指“不要出错”，而是系统在面对恶意输入、权限边界、外部副作用和组织治理要求时，仍能维持可控行为的能力。

## 2. 提示注入与上下文污染为什么危险

对 Agent 来说，提示注入的危险，不只是“模型被误导”，而是：

- 被误导后还可能继续调用工具
- 被误导后还可能修改环境
- 被误导后错误结果会继续传播

### 2.1 提示注入的本质

提示注入本质上是在争夺控制权。

它试图让系统把：

- 原本高优先级规则
- 原本可信上下文
- 原本设计好的控制边界

让位给来自外部输入、外部网页、外部文档的恶意或误导性指令。

### 2.2 上下文污染比很多人想象中更广

污染不一定都是恶意攻击。

很多时候它也可能来自：

- 过时文档
- 不可信检索结果
- 混入上下文的网页提示语
- 工具返回中的误导性文本

## 3. 为什么权限最小化是 Agent 设计的硬原则

一旦系统能调工具，权限问题就会直接决定风险上限。

最小权限原则的意义不是“保守”，而是：

> 即使系统在局部判断中出错，它也没有能力把错误放大到不可接受的程度。

### 3.1 权限不是单一开关

一个成熟系统至少会区分：

- 可读权限
- 可写权限
- 可执行权限
- 高风险不可逆权限

### 3.2 权限越粗，控制越差

如果一个 Agent 拿到的是：

- 全量读
- 全量写
- 全量执行

那么系统后面再怎么补 Prompt 和确认，风险都已经很高了。

## 4. 高风险动作为什么不能只靠模型判断

很多高风险动作在技术上是“模型也能判断要不要做”的，但这不代表控制权应该直接交给它。

典型高风险动作包括：

- 外发消息
- 改删数据
- 执行命令
- 触发对外流程
- 权限变更

### 4.1 局部最优判断不等于组织上可接受

模型可能在当前局部上下文里判断“应该发送”“应该删除”“应该执行”。

但组织角度还要问：

- 谁来承担责任
- 是否有审计
- 是否可回滚
- 是否需要人确认

### 4.2 确认机制本质上是在重新分配控制权

确认不是多余摩擦，而是：

- 把不可完全自动化的决策交还给人或上层策略

## 5. Human-in-the-loop 真正适合放在哪

不是所有步骤都值得人工确认。

更合理的做法通常是把人在回路放在：

- 高风险动作前
- 边界模糊动作前
- 关键权限切换点
- 关键结论出具前

这样既能控制风险，也不至于让系统退回纯人工工具。

## 6. 数据边界和隐私问题如何理解

Agent 一旦开始接真实业务数据，就必须面对两个问题：

- 它能看到什么
- 它会把看到的东西带到哪里

### 6.1 数据最小暴露

不是“系统能访问到什么，就都给它看”。

更好的原则是：

- 当前任务只暴露当前任务真正需要的数据

### 6.2 上下文泄漏

很多安全问题不来自工具本身，而来自上下文被错误带入。

例如：

- 前一个任务的敏感信息被带进后一个任务
- 不同用户的上下文边界没切干净

## 7. 审计与留痕为什么是治理核心

如果系统真的会做事，就必须能回答：

- 它做了什么
- 为什么做
- 用了哪些输入
- 通过哪些工具完成

没有这些信息，出了问题几乎无法追责，也无法复盘。

### 7.1 审计不只是给安全团队看

审计同时服务：

- 事故复盘
- 行为归因
- 权限核查
- 模型治理

### 7.2 留痕的重点不只是日志数量

更重要的是关键决策点能否被看见：

- 谁触发了任务
- 当前轮上下文核心是什么
- 哪个工具被用到了
- 哪个动作真正改了环境

## 8. 企业场景下的治理框架

企业环境里，Agent 的问题不会停留在“好不好用”，而会进入：

- 能不能上线
- 谁能用
- 用到什么程度
- 出问题谁负责

所以治理框架通常至少会涉及：

- 权限分层
- 高风险动作审批
- 审计与日志保存
- 数据边界
- 版本与策略管理

## 9. 常见安全反模式

### 9.1 全权限默认开放型

先把能力开满，后面再想办法补救。

这通常是最危险的起点。

### 9.2 把安全全部压给 Prompt 型

试图只靠一句“不要执行危险动作”来控制系统。

这不是设计，是许愿。

### 9.3 无审计上线型

系统可以调工具、改环境，但没有关键留痕。

这类系统一旦出问题，几乎不可复盘。

### 9.4 上下文边界混淆型

不同任务、不同用户、不同权限域之间的上下文没有切开。

这类问题既危险，也非常隐蔽。

## 10. 一个可复用的安全与治理检查框架

### 10.1 这个系统的最高风险动作是什么

先找风险上限，再谈能力开放。

### 10.2 这些动作是否已做权限分级

读、写、执行、高风险不可逆，最好明确区分。

### 10.3 关键动作有没有确认或审批点

如果没有，后续控制会很难补。

### 10.4 系统是否留下了足够审计线索

没有留痕，就没有治理能力。

> [!tip] 使用建议
> 不要问“这个 Agent 安不安全”，而要问“在什么权限、什么数据、什么动作范围内，它是否仍然可控”。

## 11. 本章应当留下的认知结论

读完这一章，你至少应该建立这些判断。

- 安全不是补丁，而是 Agent 与环境交互后的系统边界问题
- 提示注入与上下文污染，本质上是在争夺控制权
- 权限最小化是降低错误放大能力的核心手段
- 高风险动作不能只靠模型局部判断
- 审计与留痕是企业治理的基础能力

## 一页总结

- 安全不是补丁，而是 Agent 接入真实环境后的结构边界问题。
- 提示注入与上下文污染，本质是在争夺控制权。
- 权限最小化是限制错误放大的核心手段。
- 高风险动作必须重新分配控制权，不能只靠局部模型判断。
- 审计与留痕不是附属能力，而是治理基础。

## 思考练习

> [!question] 思考练习
> 选一个你熟悉的 Agent 系统，尝试回答下面的问题：
> 1. 它当前的最高风险动作是什么？
> 2. 它有没有把读、写、执行权限区分开？
> 3. 它现在的防线主要靠 Prompt、权限、确认，还是审计？
> 4. 如果明天要上线，你最担心的是提示注入、权限过宽，还是留痕不足？
